# Decision Log: Wiki Editing & Page Management

## Brainstorming Phase — 2026-03-30

### Decision: Scope — Foundation
- **Chosen:** Foundation code
- **Alternatives considered:** Throwaway spike for the editor integration
- **Rationale:** Pages are persisted to the repo. The hierarchy and ordering fields become part of every page's frontmatter — changing the data model later would require migrating all existing pages.
- **Trade-offs accepted:** Requires proper planning before coding; no quick spike shortcut.

### Decision: Editor — Decap CMS (previously agreed)
- **Chosen:** Decap CMS with direct GitHub commits
- **Alternatives considered:** Toast UI embedded, custom editor, Netlify CMS
- **Rationale:** Previously agreed in 2026-03-29 feature roadmap session. Decap handles GitHub OAuth, file creation, and markdown editing. No server required.
- **Trade-offs accepted:** Editors must have GitHub accounts with push access. OAuth proxy may be needed for GitHub Pages hosting.

### Decision: No Editorial Workflow
- **Chosen:** Direct commits to master (no PR-per-edit)
- **Alternatives considered:** Decap `publish_mode: editorial_workflow` (creates a PR per save)
- **Rationale:** Adds friction for a single-author or small-team wiki. PRs for content edits are overkill.
- **Trade-offs accepted:** No approval gate before content goes live. Can be added later via config.

### Decision: Hierarchy — Frontmatter Parent Slug (not folder structure)
- **Chosen:** `parent: slug` field in frontmatter; all pages flat under `pages/`
- **Alternatives considered:** Folder-based (original approach — `pages/a/b/c.md` infers parent from path)
- **Rationale:** Folder structure couples file organisation to wiki hierarchy. Moving a file changes its parent. Flat structure + explicit parent decouples the two, allows free reorganisation of the filesystem without breaking the tree.
- **Trade-offs accepted:** Parent references can go stale if a parent page is renamed/deleted. Build warns on unresolved parent slugs.

### Decision: Page Identity — Slug (filename stem, immutable)
- **Chosen:** Slug = filename stem (`getting-started` for `pages/getting-started.md`). Immutable after creation. Must be unique.
- **Alternatives considered:**
  - Parent reference by title — not unique, breaks on title rename
  - UUID in frontmatter — stable but opaque, ugly in frontmatter, UI friction
  - Slug derived from title (mutable) — breaks parent references on rename
- **Rationale:** Slug is already unique (filesystem enforces it for flat structure), human-readable, and naturally generated by Decap from the title. Immutability is a documented rule enforced by build hard-fail on duplicates and Decap hint text.
- **Trade-offs accepted:** Renaming a page requires also updating all `parent:` references pointing to the old slug. Rare for a wiki, acceptable cost.

### Decision: Duplicate Slugs — Build Hard-Fail
- **Chosen:** `sys.exit(1)` if two pages share the same slug
- **Alternatives considered:** Soft warning only; auto-rename to `slug-2`
- **Rationale:** Slugs are the identity key for parent references. A duplicate slug would silently corrupt the hierarchy. Hard fail forces conscious resolution.
- **Trade-offs accepted:** Build breaks until the human resolves the conflict.

### Decision: Slug Collision in Decap — Pre-Check with Suggestion
- **Chosen:** `preSave` handler checks `search_index.json`, blocks save and suggests `slug-2` if taken
- **Alternatives considered:** Only rely on build-time fail; auto-rename silently
- **Rationale:** Better to catch at edit time than fail the build after commit. Suggestion (`slug-2`) gives a clear next step. Human confirms the new slug intentionally.
- **Trade-offs accepted:** `search_index.json` may be slightly stale (last build). A very-recently-created page might not appear, but this is a rare edge case.

### Decision: Duplicate Titles — Soft Warning Only
- **Chosen:** Build warns; Decap shows an orange banner that auto-dismisses. Save is not blocked.
- **Alternatives considered:** Hard-fail on duplicate titles
- **Rationale:** Two pages can legitimately share a display title as long as their slugs differ (e.g. "Overview" in different sections). Discouraging but not banning allows flexibility.
- **Trade-offs accepted:** Users may not notice the warning if they dismiss it quickly.

### Decision: Sort Order — Frontmatter Integer Field
- **Chosen:** `sort_order: 20` in frontmatter; lower = first; default 100; alphabetical tiebreak
- **Alternatives considered:**
  - Numeric filename prefix (`01-getting-started.md`) — requires file renames to reorder
  - `.order` file per folder — extra file to maintain, folder-structure dependency
- **Rationale:** Frontmatter field is decoupled from filename and folder structure. Works with flat layout. Decap can expose it as a number input.
- **Trade-offs accepted:** Must manually maintain sort_order values; easy to end up with gaps or duplicates (resolved by tree manager page).

### Decision: Sort Order UX — Drag-and-Drop Tree Manager Page
- **Chosen:** Dedicated `/admin/tree.html` page — drag to reorder (sort_order) and reparent (parent field), batch GitHub API commit on Save
- **Alternatives considered:** Edit sort_order number directly in Decap form per-page
- **Rationale:** Editing numbers blind (without seeing the tree) is poor UX beyond ~5 pages. Drag-and-drop provides immediate visual feedback and handles both operations (reorder + reparent) in one interaction.
- **Trade-offs accepted:** Significantly more implementation complexity. GitHub API rate limits could affect large saves (unlikely for a wiki). Auth requires being logged into Decap first.

### Decision: Auto Last-Updated — Content Hash Comparison with Write-Back
- **Chosen:** Build computes SHA-256 hash of each page body. If hash differs from `content_hashes.json`, updates `last-updated` frontmatter in the source file and commits the change via GitHub Actions.
- **Alternatives considered:**
  - Always update on every build — pollutes git history with false changes
  - `git log` commit timestamp — requires git access in build, less reliable (commits can update files without changing body)
  - Only update if user explicitly sets it — defeats the purpose of automation
- **Rationale:** Hash comparison is reliable and deterministic. Write-back via GitHub Actions bot commit is the standard pattern for auto-maintained metadata.
- **Trade-offs accepted:** Build workflow needs `contents: write` permission. Bot commits add to git history. `[skip ci]` tag prevents recursive build loops.

## Planning Phase — 2026-03-30

### Decision: Architecture — Build Resolves Slugs, Not JS
- **Chosen:** `compute_hierarchy()` in `build_wiki.py` resolves `parent: slug` to a full path ID before writing `search_index.json`. The wiki JS receives a fully resolved tree — no slug resolution at runtime.
- **Alternatives considered:** Store raw slug in search_index, resolve in JS at runtime
- **Rationale:** Build is the right place for data normalisation. JS tree rendering (`buildDocTree()`) already expects `doc.parent` to be a full ID matching the `docMap` keys. Resolving at build time means zero changes to the existing tree JS.
- **Trade-offs accepted:** Unresolved parent slugs silently become null (with a build warning). This is acceptable — the page still renders, just without a parent.

### Decision: Tree.html Uses DOM API (Not innerHTML) for Security
- **Chosen:** All nodes built with `document.createElement` + `textContent`. No `innerHTML` with user-controlled data.
- **Alternatives considered:** Template literal innerHTML (simpler code)
- **Rationale:** Page titles and slugs come from user-authored content. Injecting them via innerHTML without DOMPurify is an XSS risk, even in an admin page. DOM API is equally readable and inherently safe.
- **Trade-offs accepted:** Slightly more verbose code.

### Decision: GitHub Token — Reuse Decap's localStorage Key
- **Chosen:** Read `localStorage.getItem('netlify-cms-user')` → parse JSON → extract `.token`
- **Alternatives considered:** Prompt user to enter a personal access token; use a separate OAuth flow
- **Rationale:** If the user is already logged into Decap, their token is available. Asking for a second token is redundant friction.
- **Trade-offs accepted:** Tightly coupled to Decap's internal localStorage key name. If Decap changes the key, the tree manager breaks silently. Mitigated by the "Log in via CMS Admin first" error message.

### Decision: Decap Slug Field — Hint-Based Immutability (Not Code-Enforced)
- **Chosen:** Slug field includes a visible warning hint. Build hard-fails on duplicate slugs as the final safety net.
- **Alternatives considered:** Make the slug field read-only via Decap widget config
- **Rationale:** Decap doesn't natively support a "read-only after creation" widget. A hint + build enforcement achieves the same result with less complexity.
- **Trade-offs accepted:** A user who ignores the warning can change the slug in Decap. The build will catch it only if it creates a duplicate.
